软件工程思维全程回顾
一、需求分析与功能分解（Requirements Analysis）
项目最初从一个功能清单开始——FR-1 到 FR-9。我们没有上来就写代码，而是先对需求做了结构化分解：

功能性需求（Functional）：CSV 加载、流量排序、路径搜寻、星型拓扑检测、安全规则引擎、子图导出
非功能性需求（Non-functional）：双语输出、可扩展性、日志可追溯、实时性
这体现的是软工中需求分级与优先级排序的思维——先搞清楚"要做什么"，再决定"怎么做"。

二、模块化设计（Modular Design）
整个 C++ 后端被切分成职责单一的模块：

csv_reader   → 只负责数据读取与解析
graph        → 只负责图的构建与维护
analytics    → 只负责流量统计与排序
path         → 只负责路径算法（BFS + Dijkstra）
star         → 只负责星型拓扑检测
rules        → 只负责安全规则引擎
export       → 只负责子图导出
logger       → 只负责日志系统
cli          → 只负责参数解析与命令分发
每个 .h/.cpp 文件对应一个明确的功能边界，模块间通过函数接口交互，而不是共享全局变量。这是**高内聚、低耦合（High Cohesion, Low Coupling）**原则的直接体现。

三、算法选型与权衡（Algorithm Design Trade-offs）
路径搜寻模块同时实现了两种算法：

BFS：适合无权图，时间复杂度 O(V+E)，保证最少跳数
Dijkstra：适合带权图，时间复杂度 O((V+E)logV)，保证最小拥堵代价
同时暴露了 --metric hop/congestion/both 参数，让用户在**"速度最快"和"路径最优"之间按场景选择。这是软件设计中算法策略模式（Strategy Pattern）**的应用——把算法的选择权交给调用方，而非硬编码。

四、增量构建与版本控制（Incremental Development + VCS）
整个项目从 Phase 1 到 Phase 15，每个阶段完成后立即 git commit && git push。这实践了增量交付的核心思路：每次提交都是一个可运行的状态，而不是等所有功能都完成再一次性提交。

Git 的使用也让我们在出现 bug 时有了回溯的能力——每次改动都有记录，出问题了可以定位到是哪次变更引入的。

五、多层次系统集成（System Integration）
这个项目本质上是一个多语言、多进程的集成系统：

C++ 二进制程序
    ↕ 子进程 subprocess 调用

    ↕ HTTP REST API

日志系统（

七、用户体验迭代（UX Iteration）
项目最初的"前端"只是把 C++ 的终端输出原样显示在网页里——干巴巴的等宽文本。用户反馈后，我们经历了三轮迭代：

第一轮：用正则解析文本，生成 HTML 表格（能用，但脆弱）
第二轮：C++ 后端支持 --json 输出，JS 解析 JSON 渲染（可靠，但 key 名写错导致崩溃）
第三轮：统一字段命名规范，修复 last_size 状态管理 bug，加 reset 接口
这三轮迭代正是敏捷开发（Agile） 中"构建-度量-学习"循环的缩影：快速交付一个可用版本，根据反馈发现问题，持续改进。

总体感想
这个项目最值得思考的地方，是技术债的积累方式。正则解析文本那个方案，当时能跑通，看起来是"解决了问题"，但实际上是把"前端假设 C++ 输出格式永远不变"这个不稳定的前提隐藏进了代码里。随着需求变化（双语文案、格式调整），它就会一次次爆炸。

软件工程给我们的最重要启示是：好的设计不是解决当前的问题，而是让未来的变化容易落地。 从这个角度看，JSON 协议、模块化边界、日志系统这些"多此一举"的设计，正是让系统能在几十次迭代后仍然可维护的根本原因。

软件工程思维回顾
1. 分层架构（Layered Architecture）
我们的系统被清晰地分成了三层：

层次	技术	职责
数据计算层	C++ 

./app
纯粹的算法与分析，输出结构化数据
服务层	Python FastAPI	桥接 HTTP 请求与 C++ 进程，路由分发
表现层	HTML/CSS/JS	纯展示，不含任何业务逻辑
每一层只对邻层负责，相互隔离——这正是软件工程中**关注点分离（Separation of Concerns）**的核心。

2. 接口契约的演进——从"约定格式"到"结构化 IPC"
最初前后端的"接口"是裸文本：JS 用正则去解析 C++ 的终端排版输出（空格、虚线、中英文混排）。这是一种高耦合、脆弱的隐式契约，任何一行文案改动都可能让解析崩溃。

我们最终迁移到了 --json 标志 + JSON 数组的方案。这本质上是在两个系统之间建立了一个明确的接口契约（Interface Contract）：C++ 承诺输出合法 JSON，JS 承诺按字段名读取。把隐式约定变成了显式规范。

3. 防御性编程（Defensive Programming）
Live Dashboard 的那个 last_size bug 是一个典型教训：

python
# 错误做法（攻击性编程）
live_state["last_size"] = current_size  # 先记录
fast_convert(...)                        # 再处理 → 失败了？last_size 已回不去了
# 正确做法（防御性编程）
fast_convert(...)                        # 先处理
if 成功:
    live_state["last_size"] = current_size  # 只在成功时才提交副作用
状态的更新应该"原子化"地与操作的成功绑定，不能在还未确认结果之前就修改系统状态——这是事务型设计的基本原则。

4. 可观测性（Observability）
当 Live Dashboard 一直显示"No new packets"时，你无法判断是文件没有增长、还是 last_size 卡死了。我们加了一行：

No new packets (file: 5988KB / last: 5988KB)
把系统内部状态暴露出来，让问题变得肉眼可见。这是软件工程里可观测性设计的精髓——好的系统会告诉你它当前在做什么、为什么没做。

5. 持续集成的意识——但被编译遗漏坑了
我们多次 git commit && git push，保持了良好的版本控制习惯。但"HTTPS sort 一直报错"那个 bug 根本原因是：修改了 C++ 源代码，但忘记重新 make。

这恰好说明了 CI/CD（持续集成）的价值——自动化构建可以强制保证"源代码改动 → 产物更新"这条链路不会被人为疏忽打断。在真实项目中，GitHub Actions 或 Makefile 中的依赖检查可以防止这类错误。

6. 最小化变更原则（Principle of Least Surprise）
整个重构过程中我们始终保持命令行界面不变——./app sort --top 20 在终端里仍然输出原来的人类可读格式，只有带上 --json 时才切换格式。这遵循了向后兼容的设计哲学：新功能以扩展的方式接入，不破坏已有的使用方式。

总结感想
这次开发最深的体会是：软件的可靠性不是靠"尽力而为"换来的，靠的是明确的契约、防御性的状态管理、以及系统自我暴露错误信息的能力。 一个脆弱的正则解析之所以是技术债，不是因为它"看起来丑"，而是因为它把两个系统的耦合隐藏在了人类的习惯里——某天格式变了，它就悄无声息地错了。JSON 协议不是炫技，它是让系统在变化面前依然可信赖的工程手段。
