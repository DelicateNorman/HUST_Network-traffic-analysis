1. 数据与领域模型（必须一致）

1.1 CSV 字段定义（来自任务书）

每条记录是一条“会话”（session）/通信行为： ￼
	•	Source：源 IP（字符串，如 “192.168.1.2”）
	•	Destination：目的 IP
	•	Protocol：协议号（整数：TCP=6, UDP=17, ICMP=1…）
	•	SrcPort：源端口（整数）
	•	DstPort：目的端口（整数）
	•	DataSize：流量大小（整数，单位按数据集）
	•	Duration：会话持续时间（浮点或整数，单位按数据集）

1.2 “合并边”规则（关键约束）

由于同一对 IP 可能多次通信，构建图时必须合并“源IP=Source 且 目的IP=Destination”的多条记录为一条有向边，边上累计统计： ￼
	•	total_bytes：总流量累加
	•	total_duration：总时长累加
	•	按协议分别累加（至少区分 TCP/UDP/ICMP，其它协议可放入 others）：
	•	bytes_by_proto[proto]
	•	dur_by_proto[proto]
	•	（可选）按端口/连接次数计数：session_count++

注意：合并后图中 两点之间不能存在多条边（同向）。 ￼

FR-2 构建图结构（13分）

目标：将会话数据构建为有向图：节点=IP，边=会话聚合信息。 ￼

数据结构要求（必须讲清楚）

建议邻接表；鼓励 CSR，但实现难度更高。任务书建议邻接表/CSR  ￼
最低要求：
	•	节点表：ip -> node_id
	•	邻接表：adj[u] 里是 Edge{to, stats...}
	•	边去重合并：用 unordered_map<to, EdgeStats> 或“先 hash key=(u,to)”集中合并

边属性（必须支持）
	•	total_bytes
	•	total_duration
	•	bytes_by_proto / dur_by_proto（至少能统计 TCP/UDP/ICMP）
	•	session_count

输出（控制台）
	•	图概况：
	•	节点数 N、边数 M（合并后的）
	•	输出若干节点的出边列表（用于“展示图数据结构”）
	•	（可选）输出某个 IP 的邻接信息：--show-node <ip>

验收点
	•	检查单 #2：展示图数据结构（5）+ 构建过程讲解（8） ￼

FR-3 流量排序（21分）

FR-3.1 节点总流量排序（7分）

定义：节点总流量=该 IP 作为 Source 发出的 total_bytes + 作为 Destination 收到的 total_bytes。 ￼

输出：按总流量降序列出（建议 Top-K，默认 Top 20，可参数化）。
每行：
	•	rank, ip, total_bytes, out_bytes, in_bytes, out_ratio

验收：检查单“对所有节点流量值排序输出” ￼

⸻

FR-3.2 筛选 HTTPS 节点并排序（7分）

筛选规则（任务书硬性）：
	•	协议必须 TCP：Protocol == 6
	•	目标端口必须 443：DstPort == 443
即“包含 HTTPS 连接”的节点。 ￼

注意：这里的“节点”是 IP。你需要在原始 session 或聚合边里识别满足条件的会话，然后给涉及的节点累计 HTTPS 相关流量。推荐口径：
	•	某节点 HTTPS 流量 = 该节点在满足条件的会话里作为 Source/Destination 的 DataSize 之和

输出：按“HTTPS相关流量”降序列出（Top-K）。
每行：
	•	rank, ip, https_bytes, https_out_bytes, https_in_bytes

验收：检查单“筛选包含 HTTPS 连接的节点，并排序输出” ￼

⸻

FR-3.3 筛选单向流量占比>80%节点并排序（7分）

规则（任务书硬性）：
	•	若 out_bytes / (out_bytes + in_bytes) > 0.80，则认为“单向流量占比>80%”的节点（可能端口扫描）。 ￼

输出：按总流量（out+in）降序
每行必须输出：{total_bytes, out_ratio}（检查单要求） ￼
建议行格式：
	•	rank, ip, total_bytes, out_bytes, in_bytes, out_ratio

⸻

FR-4 路径查找（21分）

目标：指定两个节点（IP），寻找：
	1.	拥塞程度最小路径（8分）
	2.	跳数最小路径（8分）
	3.	路径对比展示（5分）
￼

关键定义（任务书硬性）
	•	一条边的“拥塞度”= flow / duration（可用 total_bytes / total_duration） ￼
	•	一条路径的拥塞度：你需要明确采用一种聚合方式，并在代码里固定：
	•	建议 A（更直观）：路径拥塞度 = 路径上各边拥塞度之和（加性权重 -> Dijkstra 可做）
	•	或 建议 B（瓶颈）：max(edge_congestion)（最拥塞链路决定；用最小化最大值的算法）
	•	跳数 = 边数。例如 IP1->IP2->IP3 跳数=2。 ￼

输入
	•	CLI 参数：--src <ip> --dst <ip>
	•	或交互菜单输入

处理要求
	•	图是有向图，只能沿出边走
	•	边的 duration==0：
	•	视为不可用（跳过）或给一个极大拥塞值（推荐跳过并给提示）
	•	路径不存在要提示
	•	输出完整路径格式：IP1 -> IP2 -> ... -> IPn（任务书要求） ￼

算法建议（写死在需求里，AI 才好实现）
	•	最小跳数路径：BFS（无权图）
	•	最小拥塞路径：Dijkstra（权重=边拥塞度；路径代价=Σ拥塞度）
	•	需要恢复路径：记录 prev[node]

输出（控制台）
	•	最小跳数路径：路径字符串 + hop 数
	•	最小拥塞路径：路径字符串 + 路径拥塞总值（或瓶颈值）+ 每条边拥塞值列表（可选）
	•	路径对比：至少展示 hop 与 congestion 的差异（检查单“显示两条路径有区别”） ￼

3. 扩展功能（10分）

FR-5 星型拓扑检测（5分）

星型定义（硬性）：存在中心节点 C：
	•	C 与 ≥20 个节点相连
	•	这些外围节点只与 C 建立链接（不与其他节点建立链接） ￼

需要明确“相连”的口径（建议写死）

为了避免方向歧义，采用“无向相连”口径：
	•	若存在边 u->v 或 v->u，则认为 u 与 v 相连
	•	外围节点只允许与中心节点相连（入/出都算与中心相连）

输出格式（硬性）
	•	中心节点地址：相连节点1，相连节点2,...  ￼
（每个星型结构一行）

实现建议
	•	先构建“无向邻接集合 undirected_neighbors”
	•	对每个候选中心 C：
	•	neighbors = undirected_neighbors[C]
	•	if |neighbors|>=20
	•	检查所有 v in neighbors：undirected_neighbors[v] 只包含 C（size==1 且就是 C）
	•	输出所有满足的中心（可多个）

验收：检查单 #5  ￼

⸻

FR-6 自定义安全规则（5分）

任务书给了一个参考规则：输入 3 个 IP：ip1、ip2、ip3，其中 ip2-ip3 构成范围，表示 ip1 禁止/允许与该范围内地址建立会话，输出违反规则的会话。也允许你自定义其他规则。 ￼

本需求指定一个“必做且易验收”的规则（推荐就用任务书这个）

规则名：IP Range Block Rule
	•	输入：
	•	ip1：单个 IP
	•	ip_low, ip_high：IP 范围（闭区间）
	•	mode：deny 或 allow
	•	违规判定：
	•	若 mode=deny：任何会话中只要出现 (Source==ip1 && Destination in [low,high]) OR (Destination==ip1 && Source in [low,high]) 就算违规
	•	若 mode=allow：ip1 只允许与该范围通信，若与范围外通信则违规

关键技术点（写死，便于AI实现）
	•	IP 字符串转 32-bit 整数：a.b.c.d -> (a<<24 | b<<16 | c<<8 | d)
	•	处理范围时要支持 low>high 自动交换

输出

输出所有违规的原始 session 记录（建议打印字段全量）：
	•	Source, Destination, Protocol, SrcPort, DstPort, DataSize, Duration

验收：检查单 #5 自定义安全规则  ￼

7.3 算法约束（写死）
	•	节点总流量：out+in（基于原始 session 或基于边聚合都可，但口径必须一致）
	•	HTTPS 识别：Protocol==6 且 DstPort==443  ￼
	•	单向节点：out/(out+in) > 0.8  ￼
	•	最小跳数：BFS
	•	最小拥塞：Dijkstra，边权=bytes/duration（duration<=0 跳过）
