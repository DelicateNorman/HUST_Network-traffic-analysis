# 需求分析（黑体小2加粗居中）

（正文字体为宋体小4号，行间距固定1.5倍行距，字符间距为标准）

## 1.1 项目研发背景与动机
在当今高度数字化的世界中，网络安全问题日益严峻。无论是高校校园网、企业内部网络，还是大型数据中心，每天都会产生海量的网络流量。如何在这些庞大的网络通信数据中，快速识别出潜在的异常行为（如端口扫描、DDoS攻击预兆、非法数据渗透等），并对网络拓扑和通信瓶颈进行有效的分析，是网络管理员和安全运维人员面临的巨大挑战。

本系统“网络流量分析与异常检测系统”旨在开发一款兼具高性能底层计算内核与高级前端态势感知界面的全栈式工程项目。系统的根本目的是在杂乱无章的五元组海量通信数据包中，清洗、提取并构建有向交互拓扑图，进而支持从宏观链路分析到微观节点安全的各项核心业务。

为了贯彻**软件工程思想**，在项目初期我进行了严格的需求分析与模块化拆解。明确将系统分为：数据解析引擎、图结构建模单元、流量排行计算网关、路径寻源引擎、安全态势告警模块等。并且，为了保障代码的持续迭代和多人协同开发规范，项目自始至终采用 **Git** 进行版本控制管理，并遵循成熟的提交规范（Commit Convention）。项目全套源代码（含前后端与扩展脚本）均已托管于 **Github**：[https://github.com/DelicateNorman/HUST_Network-traffic-analysis](https://github.com/DelicateNorman/HUST_Network-traffic-analysis) 。

## 1.2 输入数据描述
系统的输入数据来源具备强大的兼容性和工程健壮性设计，主要分为两类：
1. **静态数据源 (离线 CSV 分析)**：系统需能够完全兼容 `data/network_data.csv` 给定数据集。该数据集记录了海量的网络“会话”（Session），必须处理包含 `Source`, `Destination`, `Protocol`, `SrcPort`, `DstPort`, `DataSize`, `Duration` 等 7 项核心字段的带表头 CSV。要求程序具备良好的容错机制，能够自动忽略空行及格式错误的脏数据，并在日志中记录失败行号，以确保核心进程不因脏数据而崩溃。
2. **动态抓包流提取 (pcap)**：在扩展功能边界，系统需打破离线数据的限制，引入外部脚本（基于 `Scapy`或原生接口）对物理网卡抓取到的网络数据包 `.pcap` 进行提纯降维。将二进制的网络七层协议栈数据脱水，转换为系统识别的高维统计特征输出，保证上层处理引擎无缝衔接真实态势数据。

## 1.3 输出结果描述
在输出表现层，系统被要求提供至少两种交互形态的成熟输出：
1. **统一标准的命令行输出 (CLI)**：作为核心计算进程的输出形式，所有的计算指标必须呈现出严格对齐、重点高亮、具备完备的中英双语标注的列表或日志描述字。例如输出连通子图概况、排序榜单（Top-K 排行榜）、路径对比结果和星型汇聚中心等。
2. **高频日志持久化审计 (Logs)**：后台常驻的全局线程安全日志模块需将系统的每次加载时间、模块执行状态及告警信息准确输出到指定的日志文件（如 `app.log`）中，完全贴近企业级容灾系统的审计标准。
3. **态势感知前端可视化 (GUI)**：提供极具高级感和专业性的暗黑主题网页界面（或交互大屏），能够通过按钮与图表直观呈现核心链路状态；必须支持通过绘图引擎（如 `Pyvis` 生成的力导向网络图 HTML）展现大规模拓扑节点的物理连通态，使用户能够拖动进行交互。

## 2.1 总体架构设计
在项目开发初期，针对“高性能计算”与“优秀人机交互体验”的双重需求，我摒弃了过去写作业时常用的“单体命令行程序”思路。因为单体程序难以在提供极速的底层图计算的同时，兼顾现代化的 GUI 呈现。因此，本系统引入了**微内核架构（Microkernel Architecture）**的思想，将项目设计为严格的“计算-服务-呈现”前后端分离架构，具体分为三层：

1. **核心极速计算引擎层 (C++ Backend Kernel)**：采用 C++17 编写，是整个系统算力的绝对核心。它负责处理百兆级别的流日志 IO，构建基于 HashMap 的高维内存对象拓扑网络，承担所有诸如 Dijkstra、BFS 和 Top-K 提取等 CPU 与内存密集型计算任务。
2. **轻量级通信与桥接层 (Python Middleware)**：系统不仅支持静态文件，还需要面对真实的物理网卡数据包。此层由 Python (`Scapy/RawPcapReader`) 与 `FastAPI` 构建，一方面作为高频抓包提纯器，另一方面作为 RESTful API 的微服中间件，安全地将 C++ 的运算结果映射给外部。
3. **态势感知呈现层 (Web Frontend GUI)**：不同于呆板的 Tkinter，系统采用了高级 Web 前端技术（HTML5/CSS3/JS）渲染出一个带有多维交互及暗黑毛玻璃动效的用户平台。借助它，用户不仅可以监控控制台日志的实时输出，更能与网络图进行物理碰撞交互。

## 2.2 功能模块划分
在明确了三层架构后，依据需求分析（第1章），系统被精心雕琢为以下几个正交协同的核心模块。每个模块保持高内聚低耦合，严格由独立的源文件控制（如 `graph.cpp`, `analytics.cpp`, `rules.cpp` 等）：

* **数据清洗与预处理模块 (`csv_reader.cpp`)**：系统数据的“消化道”。负责读入包含脏数据的 CSV 会话集或 Pcap 实时转化包，进行无锁的快速读取、字段校验与格式统一化。如果读到格式错误的行，不触发进程崩溃，而是记录在案后继续解析。
* **拓扑构建与驻留模块 (`graph.cpp`)**：系统的“骨架”。提取合法流数据中的 Source 和 Destination，利用常数时间复杂度的 Hash Table 实现内存寻址，并将海量连接并轨融合（聚合 Traffic Bytes 与 Duration 等），建立出高度精简版的有向带权连通图。
* **分析排序引擎模块 (`analytics.cpp`)**：系统的“数据矿工”。在构建完毕的大盘上，实现基于单节点全量流量、基于 HTTPS/TLS 安全协议、基于特定流量占比阈值（单向流量提取）的复合排序输出。
* **流量寻路探测模块 (`path.cpp`)**：系统的“导航仪”。集成并维护以最小跳数（BFS）以及以通信拥塞模型驱动的最短路径成本（Dijkstra），负责规划出图中两点间通信的最快或最可靠物理链路。
* **安防阻断与审计模块 (`rules.cpp`)**：系统的“防火墙”。遍历全图检查符合“星型”（Star Topo）多分支可疑结构的中心枢纽；执行基于自定义规则网络掩码计算（IP Subnetting）的区间会话审查。
* **系统底座支撑模块 (`logger.cpp` 等)**：全天候运行的线程级记录进程。它截获上层模块的调用结果、运行耗时以及报错预警，并按统一的日志格式（时间戳+等级+内容）写入 `out/app.log`，供工程复核。

## 3.1 数据结构设计

网络拓扑属于高度稀疏的有向图，如果采用传统的二维邻接矩阵会造成内存极度浪费并极易发生内存溢出（OOM）。故本系统核心采用了复合的高维聚类数据结构：

### 节点双向哈希映射
利用 `std::unordered_map<std::string, int>` 来实现 IP 地址到内部数组下标的 $O(1)$ 时间复杂度快速映射。对于超大规模的局域网，这种 O(1) 的寻址是非常必要的。同时，利用一个简单的 `std::vector<std::string>` 保存下标到 IP 的逆向映射用于输出。

### 极限压缩的邻接表 `Graph`
```cpp
struct Graph {
    // 高效双向映射，O(1) 复杂度根据 IP 查询内部图 ID
    std::unordered_map<std::string, int> ip_to_id;
    std::vector<std::string> id_to_ip;
    
    // 采用邻接表，极大压缩十万级稀疏网络拓扑的内存空间
    std::vector<std::vector<Edge>> adj;
};
```

## 3.2 函数实现描述

在主体功能中，我精挑细选了“最短路径与拥塞链路查询”以及“流量极速排序”作为算法设计核心，这代表了系统图论的基础能力。

### 算法1：基于 Dijkstra 的最少拥塞路径查询
由于需要寻找两点间最顺畅的通信链路，我利用链路的带宽（`total_duration / total_bytes`）作为消耗代价模型（Cost Model）。
随后采用单源最短路算法 Dijkstra。为了避免外层循环带来的大量寻找最小值消耗，这里采用带有 `std::priority_queue` 的优先队列优化。这种算法思想可以使得算法时间复杂度稳定在 $O((V+E)\log V)$，在海量连接下依然能达到极高的反馈查询速度。通过一个 `parent` 数组，在每次更新距离时记录节点的前驱，最终到达终点后回推并翻转即可获得完整的跃点路径。

### 算法2：单向异常网络嗅探搜索 (Top-K 提取)
为了从千万级别的网络参与者中，以最快速度过滤并提取单向扫描节点（即出向流量 / 总流量占比 > 0.8 的“光发不收”的主机），我们对排序算法有着特殊要求。
面对这类型局部的极值寻找需求，没有使用全局快排 `std::sort` ($O(N\log N)$)，而是结合 C++ `<algorithm>` 头文件中的 `std::partial_sort` API。该算法基于部分堆排序策略：维护一个规模为 K 的最大/最小堆，遍历剩余元素时若大于堆顶则替换，最终整个序列的前 K 个元素即为所求，且这 K 个元素保持有序。这使得系统能够以 $O(N\log K)$ 的时间复杂度击穿业务瓶颈。

## 6.1 遇到问题及解决方法

1. **大流量场景下节点遍历时间爆炸与 OOM (内存溢出) 问题**
   * **问题描述**：网络实验初期直接根据 CSV 中的原记录逐条构建有向边，造成内存中存在了几百上千万的连接实体。在执行路由寻找（Dijkstra）时遍历规模指数级上升引发崩溃。
   * **解决方法**：为了提高工程健壮性，修改了建立底层边集的逻辑点。利用 `unordered_map` Hash Map 将两个 IP 之间的同向请求动态收敛合并为单一边，用权值累加代替新连线，内存体积成百倍减少，保障了图连通遍历的常数级寻址速度。

2. **PCAP 物理抓包提取阶段的严重内存泄漏**
   * **问题描述**：为了超越任务书的要求使得产品能应用在真实的现网环境中，我编写了 `pcap_to_csv.py`。但在持续捕获网卡 `pcap` 包时，若使用基础版 Scapy `rdpcap` 直接全部读取，会导致内存在数秒内溢出。
   * **解决方法**：这敦促我学习并改用 `RawPcapReader` 配合底层二进制字节位移计算（Byte Shifting），以惰性流（Lazy Stream）读取的方式代替了一次性的大内存霸占，不仅完全规避了溢出，甚至实现了增量式的“活体”态势大屏防御机制。

## 附录2-2：Dijkstra 最低拥塞路径（path.cpp 摘要）

```cpp
// 带优先队列优化的 Dijkstra，时间复杂度 O((V+E)logV)
PathResult dijkstra_path(const Graph& g, int src, int dst) {
    int N = g.num_nodes();
    std::vector<double> dist(N, 1e18);
    std::vector<int>    parent(N, -1);
    std::priority_queue<std::pair<double,int>,
                        std::vector<std::pair<double,int>>,
                        std::greater<>> pq;  // 最小堆
    dist[src] = 0.0;
    pq.push({0.0, src});

    while (!pq.empty()) {
        auto [d, u] = pq.top(); pq.pop();
        if (d > dist[u]) continue;           // 惰性删除过期节点
        for (const auto& edge : g.adj[u]) {
            // 拥塞代价 = 持续时间 / 字节数，越大代表越拥堵
            double cost = edge.stats.total_duration /
                          std::max(1.0, (double)edge.stats.total_bytes);
            if (dist[u] + cost < dist[edge.to]) {
                dist[edge.to]   = dist[u] + cost;
                parent[edge.to] = u;         // 记录前驱，供路径回溯
                pq.push({dist[edge.to], edge.to});
            }
        }
    }
    PathResult res;
    res.found = (dist[dst] < 1e17);
    if (res.found) {
        for (int cur = dst; cur != -1; cur = parent[cur])
            res.path.push_back(cur);
        std::reverse(res.path.begin(), res.path.end());
        res.cost = dist[dst];
    }
    return res;
}
```

## 附录2-3：星型拓扑检测算法（star.cpp 摘要）

```cpp
// 正确定义：中心节点存在 >= min_leaves 个纯叶子邻居
std::vector<StarTopology> detect_stars(const Graph& g, int min_leaves) {
    int N = g.num_nodes();
    // Step1：构建无向邻居集合
    std::vector<std::unordered_set<int>> undirected(N);
    for (int u = 0; u < N; u++)
        for (const auto& e : g.adj[u]) {
            undirected[u].insert(e.to);
            undirected[e.to].insert(u);
        }

    std::vector<StarTopology> results;
    for (int c = 0; c < N; c++) {
        // Step2：筛选纯叶子子集（degree==1 表示仅连接中心节点）
        std::vector<int> leaves;
        for (int leaf : undirected[c])
            if ((int)undirected[leaf].size() == 1)
                leaves.push_back(leaf);
        if ((int)leaves.size() < min_leaves) continue;

        StarTopology st;
        st.center = g.id_to_ip[c];
        for (int leaf : leaves)
            st.leaves.push_back(g.id_to_ip[leaf]);
        std::sort(st.leaves.begin(), st.leaves.end());
        results.push_back(st);
    }
    return results;
}
```
